## The .mdc Format

Each rule file uses this frontmatter format:

```yaml
---
globs: **/directory/**     # File patterns that trigger auto-injection
alwaysApply: false         # true = always loaded; false = only when glob matches
description: Short text    # Optional human-readable purpose
---
```

Followed by markdown content.

---

## Structural Template

Based on the airweave analysis, the most effective rules follow this structure:

```
# Rule Title

## Overview
1-3 sentences: what this subsystem is and why it exists.

## Architecture
- Which files own which concerns (exact paths)
- ASCII directory tree if the subsystem has its own directory
- Component-to-file mapping table

## Key Concepts / Data Model
- Core models, their fields, and relationships
- State machines / enums if applicable
- Key types and their invariants

## Patterns & Conventions
- How to do X (with code examples)
- Naming conventions as lookup tables
- Template variables where applicable

## Common Mistakes / Gotchas
- Numbered list of likely AI errors with solutions
- ‚úÖ/‚ùå do/don't pairs where helpful
- "Why not X" explanations for non-obvious decisions

## Extension Guide
- "How to add a new Y" numbered checklist
- Which files to touch in what order

## Quick Reference
- Import paths, CLI commands, key URLs
```

Not every rule needs every section. Minimal rules (~30 lines) work for simple subsystems. Comprehensive rules (300+ lines) are appropriate for complex areas like the frontend or review workflow.

## Techniques to Use

These patterns were identified as most effective across the airweave rules:

| Technique | When to Use | Example |
|-----------|-------------|---------|
| ‚úÖ/‚ùå anti-pattern callouts | Binary do/don't rules | "‚úÖ Use `CommonViewSetMixin` / ‚ùå Never subclass `ModelViewSet` directly" |
| ASCII directory trees | Any subsystem with its own directory | Show the file layout so AI can navigate without searching |
| Named file paths for every component | Always | `django/library/models.py:Codebase` not just "the Codebase model" |
| Numbered gotchas | Complex areas with subtle errors | "1. Never call `fs.get_sip_bag()` without checking `codebase_release.bagit_info`" |
| Philosophy one-liners | Areas with design constraints | "All metadata operations are idempotent ‚Äî re-minting a DOI updates, never duplicates." |
| Pipeline/lifecycle diagrams | Multi-step processes | Review lifecycle: INVITED ‚Üí ACCEPTED ‚Üí COMPLETED |
| Self-audit checklists | Complex implementations | "Before submitting: ‚òê Models migrated ‚òê Serializer registered ‚òê URL wired" |
| Priority orderings | Resolution chains | "Permission check order: 1. Object-level (guardian) 2. Group-level (ComsesGroups) 3. Default" |
| "Human Task" labels | Operations requiring env access | Mark `manage.py` commands and Docker operations |

#### File 6: `django-vue-hybrid-rendering.mdc`
- **Maps from:** split from frontend + backend
- **Globs:** `**/templates/**,**/frontend/src/**`
- **Always apply:** `false`
- **Content guidance:**

  - Server side: Jinja2 template renders HTML with `data-*` attributes containing serialized data
  - Client side: `extractDataParams(el)` parses `data-*` into a JS object, passed as props to Vue component

  - `RootContextHTMLRenderer` adds nav, breadcrumbs, and SEO context to every HTML response
